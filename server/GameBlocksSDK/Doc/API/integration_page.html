The GBClient connects to the remote GameBlocks server to send and receive events related to one or more Game Server instances.

GameBlocks Requirements

Please read this section carefully to ensure that the GameBlocks library will work properly with your game server.

  • GBClient is a DLL and must be integrated into the server project. Win32 and x64 versions are supplied.
  • The GBClient class uses TCP non blocking connections and WSAStartup must be called before connecting via GBClient.
  • WinSock 1.1 or above is required.
  • GBClient dll/lib has been linked with Visual Studio 2009, 2010 and 2012 projects. Please let us know if you are using a different version.
  • GBClient is not thread safe.

How to Integrate

Ensure you have the connection information supplied to you by the GameBlocks team. You will need this information when integrating the GBClient dll to your codebase. The connection information includes: GameBlocks Server IP Address: The IP address used to connect the GameBlocks client to the GameBlocks Server.
GameBlocks Server Port: The Port for the GameBlocks connection.

The header file your integration will need to include is:
GBClient.h

Handle Custom Actions Sent to your GBClient
Reference the Title Configuration document for the actions which may be sent to your client:

Please refer to the Title Configuration document supplied with the integration package for information on which specific incoming actions need to be handled by the integration.

(As a note we are planning an interface to configure and review all available actions for a game title to streamline this process.)

For example if the player is banned GameBlocks will send "Kick" when the player logs or initially becomes banned. Rules can also be configured to Kick players as well and the action name, "Kick", will be the same in these cases. On incoming GameBlocks message, to your server, you must process the reserved actions with the name strings listed below. The Event Source and Player are accessible via:

const GameBlocks::GBPublicSourceId & sourceId = gClient->Incoming_GetSourceId()
const GameBlocks::GBPublicPlayerId & playerId = gClient->Incoming_GetPlayerId()

The GameBlocks Id types may be casted to int32_t, uint32_t, int_64_t, and uint64_t. Call c_str() to obtain a (const char *) representation of the Id.

The kick action has an Event pair which contains the reason associated with the kick. For example bans can be configured with reason strings which will be sent when the system kicks a banned player. This is accessible via:

const char * reason = gClient->Incoming_GetPairValueString(GB_RESERVED_EVENT_ACTION_KEY_MESSAGE);

Performance Considerations

GameBlocks sends events in a highly optimized format, however the GameBlocks Server handles events from all of the game servers and the game server packets share game packet bandwidth with GameBlocks packets. Because of this it is important to only send events at the minimum frequency required for your title's rules to produce the desired results.

As a general guideline, avoid events which are sent per frame. Aim to send less than 30 events total per frame (assuming 30fps) given the statistical probability of all supported events firing on the same frame. The Title Configuration document supplied with your integration package provides target rates for event sending.

The PSReservedEvents.h documentation describes when each reserved event should be sent.

Many important game events such as Kill, teleport, etc, only need to be sent rarely and only when the event happens.

Any performance questions please contact GameBlocks support. gbsup.nosp@m.port.nosp@m.@game.nosp@m.bloc.nosp@m.ks.co.nosp@m.m
